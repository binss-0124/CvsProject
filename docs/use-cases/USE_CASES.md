# 유스케이스 다이어그램 및 시나리오

## 🎯 시스템 개요

편의점 종합 솔루션은 고객, 점주, 본사 관리자 세 가지 주요 액터와 그들 간의 상호작용을 통해 완전한 편의점 생태계를 구성합니다.

## 👥 주요 액터 (Actors)

### 1. 고객 (Customer)
- **역할**: 상품 주문, 결제, 픽업/배송 서비스 이용
- **목표**: 편리하고 빠른 편의점 이용 경험

### 2. 점주 (Store Owner)
- **역할**: 매장 운영, 주문 처리, 재고 관리, 매출 분석
- **목표**: 효율적인 매장 운영과 수익 극대화

### 3. 본사 관리자 (Headquarters Admin)
- **역할**: 전체 시스템 관리, 점포 승인, 물류 관리, 정책 수립
- **목표**: 전체 네트워크의 효율적 운영 및 품질 관리

### 4. 시스템 (System)
- **역할**: 자동화된 처리, 알림, 데이터 분석, 결제 처리
- **목표**: 안정적이고 신뢰할 수 있는 서비스 제공

## 🔄 주요 유스케이스

### 📱 고객 관련 유스케이스

#### UC-001: 회원가입 및 로그인
```
Primary Actor: 고객
Goal: 시스템에 가입하고 로그인하여 서비스 이용

Main Success Scenario:
1. 고객이 회원가입 페이지에 접속한다
2. 이메일, 비밀번호, 개인정보를 입력한다
3. 시스템이 입력 정보를 검증한다
4. 시스템이 계정을 생성하고 프로필을 자동 생성한다
5. 고객이 인증 이메일을 확인한다
6. 고객이 로그인하여 서비스를 이용한다

Extensions:
3a. 이미 존재하는 이메일인 경우
    3a.1. 시스템이 오류 메시지를 표시한다
    3a.2. 고객이 다른 이메일을 입력한다

3b. 비밀번호가 정책에 맞지 않는 경우
    3b.1. 시스템이 비밀번호 요구사항을 안내한다
    3b.2. 고객이 올바른 비밀번호를 입력한다
```

#### UC-002: 매장 검색 및 선택
```
Primary Actor: 고객
Goal: 근처 매장을 찾고 주문할 매장을 선택

Main Success Scenario:
1. 고객이 매장 목록 페이지에 접속한다
2. 시스템이 현재 위치를 요청한다
3. 고객이 위치 권한을 허용한다
4. 시스템이 근처 매장 목록을 표시한다
5. 고객이 원하는 매장을 선택한다
6. 시스템이 매장 상세 정보를 표시한다
7. 고객이 해당 매장에서 주문을 시작한다

Extensions:
3a. 위치 권한을 거부한 경우
    3a.1. 시스템이 수동 주소 입력을 요청한다
    3a.2. 고객이 주소를 입력한다

4a. 근처에 영업 중인 매장이 없는 경우
    4a.1. 시스템이 안내 메시지를 표시한다
    4a.2. 배송 가능한 매장 목록을 제안한다
```

#### UC-003: 상품 조회 및 장바구니 담기
```
Primary Actor: 고객
Goal: 원하는 상품을 찾아 장바구니에 담기

Main Success Scenario:
1. 고객이 선택한 매장의 상품 목록을 조회한다
2. 시스템이 카테고리별 상품 목록을 표시한다
3. 고객이 카테고리를 선택하거나 상품을 검색한다
4. 고객이 원하는 상품을 선택한다
5. 시스템이 상품 상세 정보를 표시한다
6. 고객이 수량을 선택하고 장바구니에 담는다
7. 시스템이 재고를 확인하고 장바구니에 추가한다
8. 시스템이 장바구니 업데이트를 확인한다

Extensions:
6a. 재고가 부족한 경우
    6a.1. 시스템이 재고 부족 메시지를 표시한다
    6a.2. 고객이 가능한 수량으로 조정한다

6b. 다른 매장의 상품을 담으려는 경우
    6b.1. 시스템이 기존 장바구니 삭제 확인을 요청한다
    6b.2. 고객이 확인하면 새 매장으로 변경한다
```

#### UC-004: 주문 생성 및 결제
```
Primary Actor: 고객
Goal: 장바구니의 상품들로 주문을 생성하고 결제

Main Success Scenario:
1. 고객이 장바구니에서 주문하기를 선택한다
2. 시스템이 주문 확인 페이지를 표시한다
3. 고객이 픽업/배송 방식을 선택한다
4. 고객이 픽업 시간 또는 배송 주소를 입력한다
5. 고객이 쿠폰이나 포인트 사용을 선택한다
6. 시스템이 최종 금액을 계산하여 표시한다
7. 고객이 결제 방법을 선택한다
8. 시스템이 토스페이먼츠로 리다이렉트한다
9. 고객이 결제를 완료한다
10. 시스템이 주문을 생성하고 확인 번호를 발급한다
11. 시스템이 점주에게 새 주문 알림을 전송한다

Extensions:
5a. 쿠폰 사용 시
    5a.1. 시스템이 쿠폰 유효성을 검증한다
    5a.2. 유효하지 않은 경우 오류 메시지 표시

9a. 결제 실패 시
    9a.1. 시스템이 결제 실패 페이지로 이동한다
    9a.2. 고객이 다시 결제를 시도할 수 있다
```

#### UC-005: 주문 상태 추적
```
Primary Actor: 고객
Goal: 주문한 상품의 현재 상태를 확인

Main Success Scenario:
1. 고객이 주문 내역 페이지에 접속한다
2. 시스템이 고객의 주문 목록을 표시한다
3. 고객이 특정 주문을 선택한다
4. 시스템이 주문 상세 정보와 현재 상태를 표시한다
5. 시스템이 실시간으로 상태 변경을 업데이트한다

Extensions:
4a. 주문이 준비 완료된 경우
    4a.1. 시스템이 픽업 안내 알림을 표시한다
    4a.2. 매장 위치와 연락처를 제공한다

5a. 주문에 문제가 발생한 경우
    5a.1. 시스템이 문제 상황을 안내한다
    5a.2. 고객센터 연락처를 제공한다
```

### 🏪 점주 관련 유스케이스

#### UC-101: 점주 회원가입 및 매장 등록
```
Primary Actor: 점주
Goal: 시스템에 가입하고 매장을 등록하여 승인받기

Main Success Scenario:
1. 점주가 점주 회원가입 페이지에 접속한다
2. 점주가 개인정보와 매장 정보를 입력한다
3. 점주가 사업자등록증 등 필요 서류를 업로드한다
4. 시스템이 입력 정보를 검증한다
5. 시스템이 계정과 매장 정보를 생성한다
6. 시스템이 본사에 승인 요청을 전송한다
7. 본사 관리자가 서류를 검토하고 승인한다
8. 시스템이 점주에게 승인 완료 알림을 전송한다
9. 점주가 매장 운영을 시작한다

Extensions:
4a. 필수 정보가 누락된 경우
    4a.1. 시스템이 누락된 정보를 안내한다
    4a.2. 점주가 추가 정보를 입력한다

7a. 승인이 거부된 경우
    7a.1. 시스템이 거부 사유를 안내한다
    7a.2. 점주가 문제를 수정하여 재신청한다
```

#### UC-102: 주문 접수 및 처리
```
Primary Actor: 점주
Goal: 새로 들어온 주문을 확인하고 처리

Main Success Scenario:
1. 시스템이 새 주문 알림을 전송한다
2. 점주가 주문 관리 페이지에서 주문을 확인한다
3. 점주가 주문 내용과 재고를 검토한다
4. 점주가 주문을 접수 확인한다
5. 시스템이 고객에게 접수 확인 알림을 전송한다
6. 점주가 주문 상품을 준비한다
7. 점주가 주문 상태를 '준비 중'으로 변경한다
8. 준비 완료 시 점주가 상태를 '픽업 대기'로 변경한다
9. 시스템이 고객에게 준비 완료 알림을 전송한다
10. 고객이 픽업하면 점주가 '완료' 처리한다

Extensions:
3a. 재고가 부족한 경우
    3a.1. 점주가 고객에게 연락하여 상황을 설명한다
    3a.2. 대체 상품을 제안하거나 주문을 취소한다

6a. 예상보다 시간이 오래 걸리는 경우
    6a.1. 점주가 예상 시간을 업데이트한다
    6a.2. 시스템이 고객에게 지연 안내를 전송한다
```

#### UC-103: 재고 관리
```
Primary Actor: 점주
Goal: 매장의 상품 재고를 효율적으로 관리

Main Success Scenario:
1. 점주가 재고 관리 페이지에 접속한다
2. 시스템이 현재 재고 현황을 표시한다
3. 점주가 재고가 부족한 상품을 확인한다
4. 점주가 상품의 재고 수량을 업데이트한다
5. 시스템이 재고 변경 내역을 기록한다
6. 시스템이 최소 재고 미달 상품을 자동 표시한다
7. 점주가 물류 요청이 필요한 상품을 선택한다
8. 점주가 물류 요청을 생성한다

Extensions:
3a. 재고가 0인 상품이 있는 경우
    3a.1. 시스템이 해당 상품을 품절 표시한다
    3a.2. 고객에게 품절 안내를 자동 표시한다

6a. 유통기한이 임박한 상품이 있는 경우
    6a.1. 시스템이 유통기한 경고를 표시한다
    6a.2. 점주가 할인 판매를 고려한다
```

#### UC-104: 매출 분석 및 리포트 조회
```
Primary Actor: 점주
Goal: 매장의 매출 현황을 분석하고 경영 인사이트 확보

Main Success Scenario:
1. 점주가 매출 분석 페이지에 접속한다
2. 점주가 분석하고 싶은 기간을 선택한다
3. 시스템이 일별/주별/월별 매출 현황을 표시한다
4. 시스템이 상품별 판매 순위를 표시한다
5. 시스템이 시간대별 매출 패턴을 표시한다
6. 점주가 매출 추이를 분석한다
7. 점주가 인기 상품과 부진 상품을 파악한다
8. 점주가 이를 바탕으로 재고 및 운영 전략을 수립한다

Extensions:
3a. 전년 동기 대비 분석이 필요한 경우
    3a.1. 시스템이 전년 동기 데이터와 비교 분석한다
    3a.2. 성장률과 변화 추이를 표시한다

6a. 특정 상품군 분석이 필요한 경우
    6a.1. 점주가 카테고리별 필터를 적용한다
    6a.2. 시스템이 해당 카테고리만 분석하여 표시한다
```

### 🏢 본사 관리자 관련 유스케이스

#### UC-201: 매장 승인 관리
```
Primary Actor: 본사 관리자
Goal: 신규 매장 등록 신청을 검토하고 승인/거부 결정

Main Success Scenario:
1. 시스템이 새로운 매장 등록 신청 알림을 전송한다
2. 본사 관리자가 승인 대기 목록을 확인한다
3. 관리자가 특정 신청을 선택하여 상세 정보를 검토한다
4. 관리자가 제출된 서류들을 검증한다
5. 관리자가 매장 위치와 사업 계획을 평가한다
6. 관리자가 승인 또는 거부를 결정한다
7. 시스템이 결정 사항을 점주에게 알림으로 전송한다
8. 승인된 경우 매장이 활성화되어 운영을 시작할 수 있다

Extensions:
4a. 서류에 문제가 있는 경우
    4a.1. 관리자가 추가 서류 요청 메시지를 작성한다
    4a.2. 시스템이 점주에게 추가 서류 요청을 전송한다

6a. 거부하는 경우
    6a.1. 관리자가 상세한 거부 사유를 작성한다
    6a.2. 시스템이 개선 방안과 함께 거부 통지를 전송한다
```

#### UC-202: 물류 요청 승인 및 배송 관리
```
Primary Actor: 본사 관리자
Goal: 점주들의 물류 요청을 검토하고 배송을 관리

Main Success Scenario:
1. 시스템이 새로운 물류 요청 알림을 전송한다
2. 본사 관리자가 물류 요청 목록을 확인한다
3. 관리자가 각 요청의 우선순위와 긴급도를 검토한다
4. 관리자가 재고 현황과 배송 일정을 확인한다
5. 관리자가 요청을 승인하고 배송 계획을 수립한다
6. 시스템이 물류 팀에 배송 지시를 전달한다
7. 관리자가 배송 진행 상황을 모니터링한다
8. 배송 완료 시 시스템이 점주에게 도착 알림을 전송한다

Extensions:
4a. 요청 상품의 재고가 부족한 경우
    4a.1. 관리자가 부분 배송을 제안한다
    4a.2. 점주가 부분 배송에 동의하면 진행한다

7a. 배송 중 문제가 발생한 경우
    7a.1. 관리자가 지연 사유를 점주에게 안내한다
    7a.2. 새로운 배송 일정을 조정한다
```

#### UC-203: 전체 네트워크 모니터링
```
Primary Actor: 본사 관리자
Goal: 전체 매장 네트워크의 운영 현황을 모니터링하고 관리

Main Success Scenario:
1. 본사 관리자가 통합 대시보드에 접속한다
2. 시스템이 전체 매장의 실시간 현황을 표시한다
3. 관리자가 매출, 주문량, 재고 상황을 종합적으로 검토한다
4. 시스템이 이상 징후나 주의 사항을 알림으로 표시한다
5. 관리자가 문제가 있는 매장을 식별한다
6. 관리자가 해당 매장에 직접 연락하거나 지원을 제공한다
7. 관리자가 전체적인 운영 개선 방안을 검토한다

Extensions:
4a. 매출이 급격히 하락한 매장이 있는 경우
    4a.1. 관리자가 해당 매장의 상세 분석을 실시한다
    4a.2. 원인을 파악하고 개선 지원을 제공한다

5a. 시스템 장애가 감지된 경우
    5a.1. 관리자가 즉시 기술팀에 연락한다
    5a.2. 영향받는 매장들에 상황을 안내한다
```

#### UC-204: 상품 카탈로그 관리
```
Primary Actor: 본사 관리자
Goal: 전체 시스템의 상품 마스터 데이터를 관리

Main Success Scenario:
1. 본사 관리자가 상품 관리 페이지에 접속한다
2. 관리자가 새로운 상품을 등록하거나 기존 상품을 수정한다
3. 관리자가 상품 정보, 이미지, 카테고리를 설정한다
4. 관리자가 권장 판매가와 마진율을 설정한다
5. 시스템이 모든 매장에 새 상품 정보를 배포한다
6. 점주들이 새 상품을 매장에 추가할지 결정한다
7. 관리자가 상품별 전체 판매 현황을 모니터링한다

Extensions:
4a. 계절 상품인 경우
    4a.1. 관리자가 판매 기간을 설정한다
    4a.2. 시스템이 기간 종료 시 자동으로 비활성화한다

6a. 특정 상품의 판매가 부진한 경우
    6a.1. 관리자가 전체 할인 이벤트를 기획한다
    6a.2. 점주들에게 프로모션 참여를 안내한다
```

## 🔄 시스템 간 통합 유스케이스

### UC-301: 실시간 알림 시스템
```
Primary Actor: 시스템
Goal: 중요한 이벤트 발생 시 관련 사용자들에게 즉시 알림 전송

Main Success Scenario:
1. 시스템이 주요 이벤트(주문 생성, 상태 변경 등)를 감지한다
2. 시스템이 해당 이벤트의 수신 대상을 결정한다
3. 시스템이 사용자 설정에 따라 알림 방식을 선택한다
4. 시스템이 실시간으로 알림을 전송한다
5. 수신자가 알림을 확인한다
6. 시스템이 알림 확인 상태를 업데이트한다

Extensions:
3a. 사용자가 알림을 비활성화한 경우
    3a.1. 시스템이 중요도에 따라 강제 알림 여부를 결정한다
    3a.2. 매우 중요한 경우에만 알림을 전송한다

4a. 알림 전송이 실패한 경우
    4a.1. 시스템이 재전송을 시도한다
    4a.2. 여러 채널(이메일, SMS)로 대체 전송한다
```

### UC-302: 자동 재고 관리
```
Primary Actor: 시스템
Goal: 재고 변동을 자동으로 추적하고 관리

Main Success Scenario:
1. 시스템이 주문 완료 시 재고를 자동 차감한다
2. 시스템이 최소 재고량 대비 현재 재고를 비교한다
3. 재고가 부족한 경우 자동으로 점주에게 알림을 전송한다
4. 시스템이 과거 판매 패턴을 분석하여 권장 주문량을 계산한다
5. 시스템이 점주에게 권장 물류 요청을 제안한다
6. 점주가 승인하면 자동으로 물류 요청을 생성한다

Extensions:
2a. 상품이 완전히 품절된 경우
    2a.1. 시스템이 자동으로 해당 상품을 비활성화한다
    2a.2. 고객에게 품절 상태를 표시한다

4a. 계절성 상품인 경우
    4a.1. 시스템이 계절 패턴을 고려하여 주문량을 조정한다
    4a.2. 성수기와 비수기를 구분하여 계산한다
```
